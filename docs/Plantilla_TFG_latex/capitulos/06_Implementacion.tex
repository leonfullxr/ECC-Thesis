\chapter{Implementación y entorno experimental}
\label{ch:implementacion}

\section{Visión general}
\label{sec:impl-vision}

El objetivo de la parte práctica de este trabajo es construir un entorno de pruebas (\textit{benchmarking}) que permita comparar el rendimiento de RSA y ECC de forma reproducible. 
Para que la comparativa sea completa, el análisis se ha estructurado desde tres perspectivas distintas:

\begin{enumerate}
    \item \textbf{RSA frente a ECC}, evaluados a niveles de seguridad equivalentes según las recomendaciones del NIST SP 800-57 \cite{nist-sp800-57}.
    \item \textbf{Coordenadas afines frente a Jacobianas}, para cuantificar el impacto real de suprimir las inversiones modulares.
    \item \textbf{Cuerpos primos $\mathbb{F}_p$ frente a cuerpos binarios $\mathbb{F}_{2^m}$}, analizando cómo afecta la base matemática subyacente al tiempo de ejecución.
\end{enumerate}

El código se ha programado en C++17, apoyándose en la biblioteca NTL \cite{shoup-ntl} para gestionar la aritmética de precisión arbitraria y en GMP \cite{granlund-gmp} como motor de cálculo base. 
Además, para garantizar que cualquier persona pueda reproducir los resultados obtenidos sin problemas de dependencias, todo el sistema se compila y ejecuta dentro de un contenedor Docker.

A lo largo de este capítulo se detalla la arquitectura del código, las decisiones de diseño adoptadas, las herramientas utilizadas y cómo se ha validado el generador de números aleatorios empleado 
en las pruebas.

exportación de datos.


\subsection{Interfaz de línea de comandos}
\label{subsec:impl-cli}

Para interactuar con el programa, se ha diseñado una interfaz de terminal sencilla que procesa los argumentos introducidos por el usuario:

\begin{verbatim}
./bin/bench [opciones]
  -a ALGO    RSA | ECC | ECCJ | BIN | CMP   (default: RSA)
  -b BITS    Tamano de clave RSA              (default: 2048)
  -c CURVE   Nombre de curva ECC              (default: secp256k1)
  -i ITERS   Numero de iteraciones            (default: 50)
  -s SEED    fixed | random                   (default: fixed)
  -r FILE    Fichero CSV de datos brutos
  -v         Modo verboso (progreso a stderr)
\end{verbatim}

Destaca el parámetro \texttt{-a}, que permite seleccionar qué conjunto de algoritmos se desea evaluar. Se puede probar RSA, curvas primas en modo afín (\texttt{ECC}), curvas primas en modo 
Jacobiano (\texttt{ECCJ}) o curvas binarias (\texttt{BIN}). Para simplificar la recolección de datos, se creó el modo \texttt{CMP}, que ejecuta automáticamente todo el conjunto de pruebas y 
genera más de 70 mediciones individuales que cubren todos los tamaños de clave y curvas implementadas.

\section{Decisiones de diseño}
\label{sec:impl-decisiones}

\subsection{Progresión pedagógica: primero afines, luego Jacobianas}
\label{subsec:impl-progresion}

A la hora de programar las operaciones de curva elíptica, se decidió implementar primero el sistema de coordenadas afines y añadir después las Jacobianas. Esta decisión tiene un doble motivo, 
técnico y académico.

Desde el punto de vista técnico, las coordenadas afines son mucho más fáciles de verificar. La ecuación clásica de Weierstrass ($y^2 = x^3 + ax + b$) permite comprobar si un punto pertenece a la 
curva con una simple sustitución. En cambio, en coordenadas Jacobianas un mismo punto tiene infinitas representaciones válidas, lo que complica la depuración de errores. Siguiendo el principio 
de ``primero que funcione, luego optimizar'', el modelo afín sirvió como plantilla de referencia para asegurar que las matemáticas eran correctas antes de introducir complejidad.

Académicamente, programarlo en este orden permitió convertir la mejora de las coordenadas Jacobianas en un experimento medible. Como se vio en el capítulo de resultados, la aceleración 
obtenida en la práctica ronda el factor de 1,7 a 1,9, un dato que aporta mucho más valor a la memoria al poder contrastarlo con los modelos puramente teóricos.

\subsection{Implementación de SHA-256 desde cero}
\label{subsec:impl-sha256}

Aunque habría sido mucho más rápido utilizar la función SHA-256 de bibliotecas consagradas como OpenSSL, se decidió programarla desde cero siguiendo el estándar oficial (FIPS PUB 180-4). 
Esto se hizo por integridad académica, para comprender realmente cómo funciona el algoritmo de \textit{hash} que acompaña a la firma ECDSA. Además, evitar dependencias externas impide que 
posibles actualizaciones de terceros alteren los tiempos de ejecución de las pruebas.

El algoritmo opera sobre palabras de 32 bits y consta de tres fases: un \emph{preprocesamiento} (relleno del mensaje), la expansión de cada bloque en un \textit{message schedule} y una función de compresión de 64 rondas. El relleno (\textit{padding}) añade al mensaje un bit \(1\), después tantos bits \(0\) como sea necesario y, finalmente, la longitud original del mensaje en bits como un entero de 64 bits (\textit{big-endian}), de modo que la longitud total sea un múltiplo de 512 bits.

La compresión emplea seis funciones lógicas sobre palabras de 32 bits, donde \(\mathrm{ROTR}^{n}\) denota la rotación circular a la derecha, \(\mathrm{SHR}^{n}\) el desplazamiento lógico a la derecha, \(\wedge\) la conjunción bit a bit, \(\oplus\) la suma XOR y \(\lnot\) la negación:
\[
\begin{aligned}
\mathrm{Ch}(x,y,z) &= (x \wedge y) \oplus (\lnot x \wedge z), \\
\mathrm{Maj}(x,y,z) &= (x \wedge y) \oplus (x \wedge z) \oplus (y \wedge z), \\
\Sigma_0(x) &= \mathrm{ROTR}^{2}(x) \oplus \mathrm{ROTR}^{13}(x) \oplus \mathrm{ROTR}^{22}(x), \\
\Sigma_1(x) &= \mathrm{ROTR}^{6}(x) \oplus \mathrm{ROTR}^{11}(x) \oplus \mathrm{ROTR}^{25}(x), \\
\sigma_0(x) &= \mathrm{ROTR}^{7}(x) \oplus \mathrm{ROTR}^{18}(x) \oplus \mathrm{SHR}^{3}(x), \\
\sigma_1(x) &= \mathrm{ROTR}^{17}(x) \oplus \mathrm{ROTR}^{19}(x) \oplus \mathrm{SHR}^{10}(x).
\end{aligned}
\]

El Algoritmo \ref{alg:sha256} resume el cálculo del resumen una vez aplicado el relleno. Todas las sumas se realizan módulo \(2^{32}\), y \(K_0,\dots,K_{63}\) son las constantes de ronda definidas en el estándar.

\begin{algorithm}
\caption{Cálculo del resumen SHA-256 (FIPS PUB 180-4)}
\label{alg:sha256}
\begin{algorithmic}[1]
\Require Mensaje rellenado, dividido en \(N\) bloques de 512 bits \(M^{(1)},\dots,M^{(N)}\).
\Ensure Resumen \(H\) de 256 bits.
\State \((H_0,\dots,H_7) \gets\) valores iniciales (FIPS PUB 180-4)
\For{\(i = 1\) \textbf{ to } \(N\)}
  \State Dividir \(M^{(i)}\) en 16 palabras de 32 bits \(W_0,\dots,W_{15}\)
  \For{\(t = 16\) \textbf{ to } \(63\)}
    \State \(W_t \gets \sigma_1(W_{t-2}) + W_{t-7} + \sigma_0(W_{t-15}) + W_{t-16}\)
  \EndFor
  \State \((a,b,c,d,e,f,g,h) \gets (H_0,H_1,H_2,H_3,H_4,H_5,H_6,H_7)\)
  \For{\(t = 0\) \textbf{ to } \(63\)}
    \State \(T_1 \gets h + \Sigma_1(e) + \mathrm{Ch}(e,f,g) + K_t + W_t\)
    \State \(T_2 \gets \Sigma_0(a) + \mathrm{Maj}(a,b,c)\)
    \State \(h \gets g\);\quad \(g \gets f\);\quad \(f \gets e\);\quad \(e \gets d + T_1\)
    \State \(d \gets c\);\quad \(c \gets b\);\quad \(b \gets a\);\quad \(a \gets T_1 + T_2\)
  \EndFor
  \State \(H_0 \gets H_0 + a\);\ \ \dots\ \ \(H_7 \gets H_7 + h\)
\EndFor
\State \Return \(H \gets H_0 \,\Vert\, H_1 \,\Vert\, \cdots \,\Vert\, H_7\)
\end{algorithmic}
\end{algorithm}

\subsection{Uso de curvas estandarizadas}
\label{subsec:impl-curvas-estandar}

El proyecto no genera curvas elípticas aleatorias propias, sino que utiliza parámetros fijos estandarizados por el NIST \cite{nist-fips186-4} y el grupo SECG \cite{sec2-v2}. 

Generar curvas seguras desde cero requiere algoritmos muy complejos (como el de Schoof-Elkies-Atkin) que exceden los objetivos de este TFG. Al usar parámetros validados por la comunidad 
criptográfica, nos aseguramos de estar trabajando con curvas matemáticamente robustas. Esto también nos permite realizar una comparación justa y orientativa de nuestros resultados frente a 
bibliotecas reales que usan estos mismos estándares.

Aun así, por buenas prácticas, el código incluye funciones de validación (\texttt{CurveParams::validate()}) que comprueban en tiempo de ejecución las propiedades básicas de los parámetros 
introducidos, como asegurar que el punto generador realmente pertenece a la curva o que el discriminante no es cero. Se omiten comprobaciones más pesadas (como la primalidad del orden $n$) 
porque se asume la fiabilidad de las curvas publicadas oficialmente.

\section{Generador de números aleatorios}
\label{sec:impl-rng}

\subsection{Diseño}
\label{subsec:impl-rng-diseno}

En criptografía, un mal generador de números aleatorios arruina la seguridad de cualquier algoritmo. Para gestionarlo correctamente, se programó una interfaz común que permite generar 
valores, derivando el trabajo real a las funciones nativas de NTL.

La característica más importante de este módulo es que permite fijar una semilla inicial. Si ejecutamos el programa con una semilla fija (por ejemplo, \texttt{-s fixed}), la secuencia de 
números generados será exactamente la misma cada vez que iniciemos el sistema. Esto es vital para depurar errores, pero sobre todo para medir el rendimiento, ya que, nos asegura que estamos 
comparando diferentes algoritmos frente a los mismos valores aleatorios matemáticos, haciendo que la comparativa sea justa y totalmente reproducible. Para pruebas reales se puede activar el modo 
aleatorio, que toma el tiempo del sistema como semilla dinámica.

\subsection{Validación estadística}
\label{subsec:impl-rng-validacion}

Para comprobar que el generador no produce patrones predecibles, se programó un módulo de evaluación apoyado en rutinas de Python. Se comprobó la uniformidad de los números, se realizaron 
pruebas de rachas (para evitar secuencias repetitivas de bits) y se midió la autocorrelación y la entropía. Todos estos controles estadísticos superaron los umbrales de validación, 
confirmando que la salida de NTL se comporta de forma adecuada para el alcance de este trabajo.

\section{Motor de medición de tiempos}
\label{sec:impl-benchmark}

\subsection{Metodología}
\label{subsec:impl-metodologia}

Para que las medidas de tiempo sean precisas, el sistema de evaluación sigue una rutina estricta para cada algoritmo probado. Primero, realiza tres ejecuciones de ``calentamiento'' 
que no se miden. Esto sirve para cargar las instrucciones en la memoria caché del procesador. A continuación, ejecuta la operación el número de veces solicitado por el usuario, 
cronometrando cada iteración con precisión de microsegundos mediante la librería estándar de C++ (\texttt{std::chrono}). Por último, calcula automáticamente la media, mediana y desviación 
de los datos recogidos.

Para aplicar esto a funciones tan distintas como firmar con RSA o derivar un secreto con curvas elípticas, el cronómetro se programó mediante un sistema de plantillas (\textit{templates}) en C++. 
Esto permite encapsular cualquier fragmento de código y medirlo utilizando exactamente el mismo mecanismo, evitando que el sistema de recolección de datos introduzca retrasos desiguales.

Destaca el parámetro $a$ de sus ecuaciones. Las curvas del NIST emplean $a = -3$, lo cual no es casualidad, porque permite reescribir y simplificar parte de las fórmulas de duplicación en

\section{Limitaciones del prototipo}
\label{sec:impl-limitaciones}

Para concluir la descripción técnica, es esencial dejar claro que el código desarrollado es una herramienta educativa de investigación. Aunque funciona correctamente desde el plano matemático, 
no está preparado para integrarse en una aplicación de seguridad real debido a las siguientes limitaciones:

\begin{enumerate}
    \item \textbf{Sin protección contra canales laterales:} Los algoritmos programados (como el tradicional \textit{double-and-add}) ejecutan distintas secuencias lógicas dependiendo de los bits de la clave privada. Un atacante podría observar el consumo de energía o los retrasos del procesador para deducir dicha clave. Los sistemas comerciales usan rutinas de ``tiempo constante'' para prevenir esto.
    \item \textbf{Riesgo en las firmas (\textit{nonces}):} Al firmar con ECDSA, se genera un número aleatorio de un solo uso (\textit{nonce}). Si el generador fallara y se repitiera este número, la clave privada quedaría expuesta. Los sistemas modernos mitigan esto generando el \textit{nonce} de forma determinista (RFC 6979), algo que no se ha integrado en este prototipo.
    \item \textbf{Derivación de secretos básica:} Al intercambiar claves con ECDH, el programa simplemente toma la coordenada $X$ del punto resultante como secreto compartido. Un sistema seguro pasaría este valor por una función de derivación de claves robusta (KDF).
    \item \textbf{RSA básico:} La implementación usa el algoritmo RSA en su forma básica (\textit{textbook RSA}). Para ser seguro contra manipulaciones, requeriría añadir esquemas de relleno (como OAEP o PSS).
    \item \textbf{Validación de curvas asumida:} Confiamos en que las curvas proporcionadas por el NIST y el SECG son seguras, omitiendo verificaciones muy costosas computacionalmente sobre los parámetros proporcionados.
\end{enumerate}

\noindent Estas carencias son habituales en proyectos académicos y no invalidan las métricas de rendimiento recogidas, pero ilustran la distancia real entre una implementación matemática correcta y un \textit{software} criptográfico seguro.
